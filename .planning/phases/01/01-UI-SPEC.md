## Color

Color system uses oklch CSS variables defined in `website/src/styles/theme.css` with light/dark mode support. All values are inherited from the shadcn slate base preset.

| Role | Value (Light) | Value (Dark) | Usage |
|------|---------------|--------------|-------|
| Dominant (60%) | `oklch(1 0 0)` -- white | `oklch(0.129 0.042 264.695)` -- near-black | Background, surfaces, auth page background |
| Secondary (30%) | `oklch(0.968 0.007 247.896)` -- light slate | `oklch(0.279 0.041 260.031)` -- dark slate | Cards, sidebar, nav, muted areas |
| Accent (10%) | `oklch(0.208 0.042 265.755)` -- dark slate (primary) | `oklch(0.929 0.013 255.508)` -- light slate (primary) | Auth submit buttons, active nav items, selected states |
| Destructive | `oklch(0.577 0.245 27.325)` -- red | `oklch(0.704 0.191 22.216)` -- lighter red | Error messages, form validation errors, danger zone actions |
| Muted foreground | `oklch(0.554 0.046 257.417)` -- mid slate | `oklch(0.704 0.04 256.788)` -- lighter slate | Descriptions, placeholder text, secondary info |

### Accent Reserved For

The `primary` color token is reserved exclusively for:
1. Auth form submit button ("Sign in", "Log in")
2. Active navigation item highlighting (Phase 2+)
3. Primary action buttons in confirmation dialogs

The `accent` color token is reserved for:
1. Hover state on ghost/outline buttons
2. Sidebar active item background
3. Keyboard focus ring

## Phase-Specific Interaction Contracts

### Login Flow (All Portals)

1. **Form validation**: Client-side validation via Zod schemas matching OpenAPI request types (LoginRequest, PortalLoginRequest, AgentPortalLoginRequest)
2. **Loading state**: Submit button shows Loader2 spinner (from lucide-react), text changes to "Signing in..." / "Logging in...", button disabled
3. **Error display**: API error messages shown inline via FormMessage (field-level) and Sonner toast (form-level)
4. **Success redirect**: On successful login, navigate to the portal's default route (replace: true to prevent back-to-login)
5. **Password field**: Uses existing PasswordInput component (eye toggle)

### Token Refresh (Silent)

1. **Interceptor**: Axios response interceptor catches 401 errors
2. **Refresh attempt**: Calls /auth/refresh with stored refresh token
3. **Retry**: On success, retries the original request transparently
4. **Failure**: On failure, clears stored tokens, redirects to login with no user-visible error (toast: "Your session has expired. Please sign in again.")

### Change Password

1. **Three fields**: Current password, new password, confirm new password
2. **Validation**: Zod schema enforces min 8 chars for new password, password confirmation match
3. **Success**: Toast notification + redirect to dashboard after 1.5s delay
4. **Error**: Inline field errors for validation, toast for API errors (wrong current password)

### Logout

1. **Confirmation**: AlertDialog with "Sign Out" title and confirmation message
2. **Action**: Calls POST /auth/logout to blacklist token server-side
3. **Cleanup**: Clears Zustand store, localStorage/sessionStorage tokens
4. **Redirect**: Navigate to login page (replace: true)

### Route Guards (NAV-04)

1. **Three route groups**: `(auth)` for unauthenticated pages, `_authenticated` for admin, portal routes for customer/agent
2. **Check**: Before render, verify auth store has valid user + non-expired token
3. **Redirect**: Unauthenticated users on protected routes redirect to their portal's login page
4. **Expired tokens**: Trigger silent refresh; if refresh fails, redirect to login

### Auth Layout (Shared)

Reuse existing `AuthLayout` pattern:
- Full-viewport centered grid (`h-svh`)
- Max-width card container (`sm:w-[480px]`)
- Logo + title at top
- Card with header (title + description), content (form), optional footer
- Responsive: full-width on mobile, centered card on desktop
