## Spacing Scale

Tailwind CSS v4 default spacing scale (multiples of 4). Same as Phase 1 -- no changes.

| Token | Value | Usage |
|-------|-------|-------|
| 1 | 4px | Icon gaps, inline padding |
| 2 | 8px | Compact element spacing |
| 3 | 12px | Input padding, small gaps |
| 4 | 16px | Default element spacing, form field gaps |
| 6 | 24px | Section padding, card content padding |
| 8 | 32px | Layout gaps, card inner padding |
| gap-4 | 16px | Dashboard widget grid gap |
| gap-6 | 24px | Section-to-section gap |

Exceptions: Dashboard widget grid uses `gap-4` (16px) for tight visual grouping of KPI cards. Data table rows use default padding from shadcn table component. Source: design decision for dense information display.

## Typography

Same scale as Phase 1, consolidated for Phase 2 widget needs. Exactly 4 sizes, 2 weights.

| Role | Size | Weight | Line Height | Source |
|------|------|--------|-------------|--------|
| Label | 14px (text-sm) | 600 (semibold) | none (leading-none) | Inherited from Phase 1; weight adjusted to 600; also used for KPI card titles (Widget Label merged here) |
| Body | 14px (text-sm) | 400 (regular) | 1.5 | Inherited from Phase 1; also used for trend text in KPI cards (previously 12px text-xs, now 14px with `text-muted-foreground`) |
| Card Title | 18px (text-lg) | 600 (semibold) | 1.0 (leading-none font-semibold) | Inherited from Phase 1 |
| Heading | 20px (text-xl) | 600 (semibold) | 1.2 (tracking-tight) | Inherited from Phase 1, page headings; weight adjusted to 600 |
| Display | 24px (text-2xl) | 600 (semibold) | 1.0 | NEW for Phase 2 -- KPI metric numbers (Widget Value consolidated here); weight 600 instead of 700 |

**Weight policy:** 400 (regular) for body text and secondary content. 600 (semibold) for all headings, labels, titles, and metric values. No 500 or 700 weights used in this phase.

### Widget Typography Pattern

Dashboard KPI cards follow this pattern:
- Card title: `text-sm font-semibold text-muted-foreground` (14px, 600) -- reuses Label role
- Metric value: `text-2xl font-semibold` (24px, 600) -- reuses Display role
- Trend text: `text-sm text-muted-foreground` (14px, 400) -- reuses Body role with muted color (previously text-xs, merged to avoid a 5th size)

## Color

Color system unchanged from Phase 1. All values from oklch CSS variables in `website/src/styles/theme.css`.

| Role | Value (Light) | Value (Dark) | Usage |
|------|---------------|--------------|-------|
| Dominant (60%) | `oklch(1 0 0)` -- white | `oklch(0.129 0.042 264.695)` -- near-black | Background, surfaces, main content area |
| Secondary (30%) | `oklch(0.968 0.007 247.896)` -- light slate | `oklch(0.279 0.041 260.031)` -- dark slate | Cards, sidebar, muted areas, table rows |
| Accent (10%) | `oklch(0.208 0.042 265.755)` -- dark slate (primary) | `oklch(0.929 0.013 255.508)` -- light slate (primary) | Active nav items, primary CTAs |
| Destructive | `oklch(0.577 0.245 27.325)` -- red | `oklch(0.704 0.191 22.216)` -- lighter red | Deactivate user actions, error states |
| Muted foreground | `oklch(0.554 0.046 257.417)` -- mid slate | `oklch(0.704 0.04 256.788)` -- lighter slate | Secondary info, trend text, timestamps |

### Phase 2 Status Colors (Semantic Badges)

Router and user status badges use semantic color mapping:

| Status | Badge Variant | Foreground | Background |
|--------|---------------|------------|------------|
| Online | `default` (primary) | primary-foreground | primary |
| Offline | `destructive` | destructive-foreground | destructive |
| Unknown / Syncing | `secondary` (muted) | muted-foreground | secondary/muted |
| Active | `default` (primary) | primary-foreground | primary |
| Inactive / Deactivated | `secondary` (outline) | muted-foreground | transparent, border muted |

### Accent Reserved For

Inherited from Phase 1. The `primary` color token is reserved for:
1. Submit buttons in create/edit forms
2. Active navigation item highlighting in sidebar
3. Primary action buttons in confirmation dialogs
4. Router status "online" badge (new for Phase 2)

The `accent` color token is reserved for:
1. Hover state on ghost/outline buttons
2. Sidebar active item background
3. Keyboard focus ring

### Chart Colors

Dashboard charts use the existing `chart-1` through `chart-5` CSS variables defined in theme.css. Do NOT introduce new chart colors.

## Phase-Specific Interaction Contracts

### Router Selector (NAV-01, NAV-02)

1. **Location**: Replace `TeamSwitcher` in `SidebarHeader` with a new `RouterSelector` component
2. **Layout**: Uses the same `SidebarMenuButton size="lg"` pattern as the existing TeamSwitcher -- router icon + name + status badge + chevron
3. **Dropdown**: Uses shadcn `Select` component (not dropdown-menu) for better keyboard navigation and search
4. **Data source**: `GET /api/v1/routers` for the list, `GET /api/v1/routers/selected` for initial selection
5. **Selection**: On select, call `POST /api/v1/routers/select/{id}` and update Zustand store
6. **Status badge**: Each router item shows a small `Badge` with semantic color:
   - `online` -- default variant (primary)
   - `offline` -- destructive variant
   - `unknown` -- secondary variant (outline)
7. **No router state**: When no routers exist, show "No routers" text with a link/button to add one
8. **Persistent**: Selected router ID stored in Zustand (or localStorage) to persist across refreshes

### Real-Time Ping Display (NAV-03)

1. **Location**: Right side of `Header` component, after any existing content (ms-auto)
2. **WebSocket**: Connect to `GET /api/v1/routers/{router_id}/monitor/ws/ping?address=8.8.8.8`
3. **Display format**: `{value} ms` with `Activity` icon from lucide-react
4. **Update frequency**: Display the latest latency value from the WebSocket stream
5. **States**:
   - Connected + receiving: show `{latency} ms` in muted-foreground
   - Connecting: show "..." skeleton pulse
   - No router selected: show "-- ms" in muted-foreground
   - Connection lost: show "timeout" in destructive color
   - Router offline: show "offline" in destructive color (do not attempt WS connection)
6. **Cleanup**: Disconnect WebSocket on unmount and when selected router changes
7. **Visual treatment**: Small, non-intrusive -- uses `text-sm text-muted-foreground` with an icon. Not a badge, not a card.

### Sidebar Navigation Groups (NAV-05)

1. **Restructure**: Replace the current template sidebar data (`sidebar-data.ts`) with ISP domain groups
2. **Pattern**: Use existing `NavGroup` component with `title` and `items` array
3. **Disabled items**: Items for unimplemented phases use `disabled: true` and lower opacity
4. **Icons**: Each nav item uses a lucide-react icon. Defined in the sidebar data file.
5. **MikMongo branding**: Replace "Shadcn Admin" with "MikMongo" and subtitle "ISP Management"
6. **Footer**: Keep existing `NavUser` (already reads from auth store)

### Dashboard Overview Widgets (DASH-01)

1. **Layout**: 4-column responsive grid (`grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-4 gap-4`)
2. **Widget pattern**: Each widget is a `Card` with:
   - `CardHeader`: flex-row with title (text-sm font-semibold text-muted-foreground) + icon (text-muted-foreground, 16x16)
   - `CardContent`: metric value (text-2xl font-semibold) + trend text (text-sm text-muted-foreground)
3. **Data source**: `GET /api/v1/reports/summary` for revenue/invoices data. Customer and subscription counts from the same endpoint or derived.
4. **Loading state**: `Skeleton` component replacing metric value and trend text
5. **Currency formatting**: Use `Rp` prefix with Indonesian number formatting (e.g., "Rp 15.230.000")
6. **Chart**: Keep the existing `Overview` bar chart (from Recharts) but wire it to actual revenue data from the report API
7. **No tabs**: Remove the Overview/Analytics/Reports tabs. Single page with all widgets visible.

### Router Health Cards (DASH-02)

1. **Location**: Below KPI widgets in a 3-column grid or horizontal scroll
2. **Pattern**: Each router is a `Card` showing:
   - Router name + area (CardTitle)
   - Status badge (online/offline/unknown)
   - Last seen timestamp (muted-foreground)
3. **Data source**: `GET /api/v1/routers` (same list as sidebar selector)
4. **Interactive**: Clicking a router card selects it (same as sidebar selector)
5. **Empty state**: "No Routers Configured" card with "Add Router" button
6. **Maximum display**: Show up to 6 router cards. If more, show "View all {n} routers" link to the routers page (Phase 3).

### Recent Activity Feed (DASH-03)

1. **Location**: Below router health section, right column or full width
2. **Layout**: Simple list of activity items inside a `Card`
3. **Item pattern**: Each item shows:
   - Activity description (e.g., "New customer registered", "Payment received")
   - Relative timestamp (e.g., "2 min ago", "1 hour ago")
   - Optional entity link (clickable)
4. **Data source**: This is a new data requirement. If the API does not provide an activity feed endpoint, display the 5 most recent items from combined data (latest users, payments, etc.) fetched individually, or show a placeholder "Activity feed coming soon" message.
5. **Empty state**: "Activity will appear here as you manage your ISP."
6. **Loading state**: 5 skeleton rows

### Admin User List (USER-01)

1. **Layout**: Full-width `Table` using `@tanstack/react-table` (already installed)
2. **Columns**:
   - Name (avatar with initials + full_name) -- primary, sortable
   - Email -- sortable
   - Role (badge with role display name) -- filterable
   - Status (active/inactive badge) -- filterable
   - Last Login (relative timestamp or "Never")
   - Actions (dropdown menu: Edit, Deactivate/Activate, Delete)
3. **Pagination**: Offset-based via `limit`/`offset` query params from OpenAPI. Show page controls at bottom.
4. **Search**: Text input above table, client-side debounce (300ms), filters by name or email
5. **Role filter**: `Select` dropdown above table, filters by role enum
6. **Data source**: `GET /api/v1/users?limit={}&offset={}`
7. **Loading state**: `Skeleton` rows matching table column layout
8. **Row actions**: `DropdownMenu` with trigger (MoreHorizontal icon from lucide-react):
   - Edit -- opens edit dialog
   - Deactivate / Activate -- shows AlertDialog confirmation
   - Delete -- shows destructive AlertDialog confirmation (only for non-superadmin self)
9. **Self-protection**: Current user cannot deactivate or delete themselves. Hide those actions for the logged-in user's row.

### Create User (USER-02)

1. **Trigger**: "Add User" button in page header (primary variant)
2. **Container**: `Dialog` with form inside
3. **Fields**: Full Name, Email, Phone (optional), Role (Select), Password, Confirm Password
4. **Validation**: Zod schema matching `CreateUserRequest` from OpenAPI
   - full_name: required string
   - email: required, valid email format
   - phone: optional string
   - role: required, one of [superadmin, admin, cs, billing, technician, readonly]
   - password: required, min 8 chars
   - confirm_password: required, must match password
5. **Submit**: `POST /api/v1/users`
6. **On success**: Close dialog, show toast, refresh table
7. **On error**: Show field-level errors via FormMessage, form-level errors via toast

### Edit User (USER-03)

1. **Trigger**: "Edit" from row actions dropdown
2. **Container**: `Dialog` with pre-filled form
3. **Fields**: Same as create EXCEPT password fields are absent (password change is a separate feature from Phase 1)
4. **Pre-fill**: Fetch user data from `GET /api/v1/users/{id}` and populate form
5. **Validation**: Same as create minus password fields
6. **Submit**: `PUT /api/v1/users/{id}` (note: OpenAPI shows delete but no explicit PUT -- confirm with API; if no PUT exists, use POST or the appropriate method)
7. **On success**: Close dialog, show toast, refresh table row

### Deactivate/Delete User (USER-04)

1. **Deactivate**: AlertDialog with destructive styling
   - Title: "Deactivate User"
   - Body: "Are you sure you want to deactivate {user_name}? They will no longer be able to log in."
   - Confirm: destructive variant button "Deactivate"
   - Action: `DELETE /api/v1/users/{id}` (API uses DELETE -- deactivation is effectively deletion)
2. **Self-protection**: Hide deactivate/delete actions for the current user's row
3. **On success**: Show toast, remove row from table or update status badge
